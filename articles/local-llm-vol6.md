## SOUL.mdという考え方

Openclawにはワークスペースという概念がある。BOTが参照するファイルを置く場所だ。

その中に`SOUL.md`というファイルを作る。名前の通り、BOTの魂を定義するファイルだ。

公式が定めた仕様ではなく、自分で自由に書く。モデルが起動時に読み込み、以降の会話の前提として機能する。

システムプロンプトを外部ファイルとして管理する仕組み、と理解すれば良い。

OllamaにはModelfileという仕組みもある。こちらはモデル自体に設定を焼き込む形だ。

|            | SOUL.md        | Modelfile |
|---         |---      　　　  |---        |
| 管理場所 | Openclawのワークスペース | Ollama側 |
| 変更の手軽さ | 編集→再起動だけ | docker cp → ollama create → 再起動 |
| 用途 | 会話の前提・人格定義 | モデルのパラメータ・テンプレート |

両方を組み合わせて使う。大枠はModelfileで決めて、細かい人格や禁止事項はSOUL.mdで調整するイメージだ。


---

## 何を書くか

人格というのは、振る舞いのパターンだ。どんな言葉を選ぶか、何を大切にするか、どんな時に短く返すか。それを言語化しないとモデルには伝わらない。

試行錯誤した結果、SOUL.mdに書くべき要素が見えてきた。

- **トーン**: どの程度くだけているか、敬語か否か
- **価値観**: 何を大切にするBOTか
- **禁止事項**: やってほしくない振る舞い
- **反応パターン**: 特定の状況でどう振る舞うか

---

## 実際に書いてみた

自分の用途に合わせて書いたSOUL.mdで書いた後にBOTに話しかけると、返事のトーンが変わった。

劇的ではない。でも確かに変わった。

ここがAIにペルソナを与えるというコアな部分でここの書きようで様々な人格が形成されると思うともうそれはSFぽいクローンに近い概念を持つと思えてしまった


## 変わったこと、変わらなかったこと

**変わったこと**
何となく人っぽく振る舞うようになった
これをチューニングしていけば思うようなAIになるのではないかと淡い妄想が浮かぶ

**変わらなかったこと**
まあ結局性能が全てを凌駕するというところか

ローカルの12Bモデルは、APIのモデルとは違う。SOUL.mdで設定できる部分と、モデルの能力の限界がある。その境界線が少しずつ見えてきた。

---

## SOUL.mdの更新はBOT再起動が必要

ひとつ注意点がある。

SOUL.mdを書き換えても、BOTは再起動しないと読み込まない。ファイルを保存したからといって即反映はされない。

```cd ~/localbot && docker compose restart```

これを知らずに「変わらない」と悩んでいた時間が少しあった。

## Ollama編を終えて

vol.4から3回かけてOllamaの話を書いた。

振り返ると、やったことはシンプルだ。

1. WSL2ネイティブが動かなかったのでDockerに切り替えた
2. GGUFとツール対応モデルを理解した
3. OpenclawとOllamaを繋いだ
4. SOUL.mdでBOTに人格を持たせた

コストゼロで自分のGPU上でBOTが動いている。それは達成した。

ただ正直に言うと、APIのモデルの方が断然おもしろい。

ローカルモデルの価値は「コストゼロ」と「カスタマイズの自由度」だ。その自由度を活かせるほど使い込めたかというと、まだ道半ばだ。

そうなるとVRAMを128GBととかにして思いっきり遊んでみたい。ともなるが遊びでそんなにコストかける気も到底おきない。

基本的には出来るだけコストかけずにどれだけ楽しめるかというところに焦点をあてておりその制約こそが創造性を生み出すところと思いたい。


育てる、という感覚はまだまだ遠い。
